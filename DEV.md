# 开发规范

本项目需要长期跟随上游演进。为了在合并上游更新时减少冲突，新增功能、修复和本地业务扩展必须尽量保持“可识别、可隔离、可删除、可重放”。

## 1. 修改前判断

只做有明确业务价值的修改：

- 新增真实业务能力。
- 修复真实线上或可复现 bug。
- 影响鉴权、权限、计费、额度、订阅、幂等、状态流转、安全、数据一致性的行为。
- 当前代码仍然使用或依赖的能力。

避免以下修改：

- 临时调试代码。
- 无业务目的的格式化、重排、重命名。
- 和当前任务无关的重构。
- 恢复已废弃入口、旧路由、旧交互或旧兼容层。
- 已被当前实现覆盖的历史兼容补丁。

## 2. 最小化上游冲突

### 2.1 新逻辑优先放新文件

能新增文件承载的业务逻辑，不要塞进上游原文件。新增文件的第一行使用本地标记：

```text
sudoapi: <English description.>
```

同一个业务能力使用完全相同的 description，并以英文句号结尾。

Go 文件示例：

```go
// sudoapi: Official Seedance task adaptor.

package doubao
```

TypeScript 文件示例：

```ts
// sudoapi: Official Seedance channel metadata.

export const seedanceChannelType = 10001
```

Markdown / SQL 等文件按对应注释语法使用相同标记。

### 2.2 现有文件只放集成点

现有文件指来自上游、第一行没有 `sudoapi` 标记的文件。修改现有文件时只放必要集成点，例如：

- 注册 adaptor。
- 添加路由。
- 添加 channel type 映射。
- 添加配置热更新 hook。
- 添加一个清晰的分支入口。

不要在现有文件中放大段业务实现。只做最小注册/分发接入。

### 2.3 插入位置

为了减少未来合并冲突：

- 新增路由放在相关 group 的末尾。
- 新增 helper 尽量放在文件末尾。
- 新增 struct/interface 字段尽量放末尾。
- 新增常量优先放独立 const 块；必要时和上游常量块之间保留空行，避免 `gofmt` 重新对齐上游代码。
- 新增对象字段尽量放对象末尾，并和对应结构体顺序一致。
- 多处修改尽量合并成连续 diff 块。
- 不为了美观调整上游代码的导入、空行、顺序或命名。

## 3. 后端实现约束

- JSON marshal/unmarshal 必须使用 `common` 包封装函数，遵守 `AGENTS.md`。
- 数据库改动必须兼容 SQLite、MySQL、PostgreSQL。
- 请求 DTO 的可选标量字段使用 pointer + `omitempty`，保留显式零值。
- 计费、额度、订阅、日志相关逻辑必须有可读的回归测试或明确验证说明。
- 不修改 `GOCACHE`，使用 Go 默认缓存目录。

## 4. 前端实现约束

- 只在确实需要用户界面时改前端。
- 默认前端是 `web/default`，使用 Bun。
- 用户可见文本必须走 i18n。
- 同步上游时，不要为了本地后端功能随手整理前端旧代码。

## 5. 测试和验证

根据影响面运行最小但有效的测试：

- 后端 service/model 行为：运行目标 package 和目标 `-run`。
- handler/API 行为：运行 controller/router 相关 package。
- 计费逻辑：覆盖价格、分组倍率、预扣、补扣、退款、零价/缺价。
- 跨数据库改动：至少说明 SQLite/MySQL/PostgreSQL 兼容点。

测试失败时区分：

- 本次改动引入的问题，必须修。
- 仓库既有环境问题或无关失败，最终说明具体失败项。

## 6. 给 AI 代理的要求

AI 修改代码前必须先读 `AGENTS.md` 和本文件。做本地业务增强时，优先选择低冲突结构：

1. 新文件承载主要逻辑。
2. 现有文件只放最小接入点。
3. 最终说明修改点、验证命令、已知无关失败。
